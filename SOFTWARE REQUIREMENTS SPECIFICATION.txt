1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the functional and non-functional requirements for the Adaptive Vendor Relevance Engine (AVRE). AVRE is a smart web platform designed to match urgent resource requests from users with the most relevant nearby vendors using an adaptive multi-factor scoring engine. This document is intended for developers, designers, project managers, and stakeholders involved in the design, development, and deployment of the AVRE system.

1.2 Project Scope
AVRE addresses a critical gap in emergency resource distribution. When individuals urgently require resources such as medicines, food, oxygen cylinders, or other emergency supplies, they currently rely on manual searches and random availability checks. AVRE automates this process by:
•	Accepting structured resource requests from users
•	Evaluating all registered vendors against the request
•	Ranking vendors using a weighted multi-factor relevance score
•	Displaying the top-matched vendors to the user in real time
•	Facilitating request confirmation between user and vendor

2. Overall Description
2.1 Product Perspective
AVRE is a standalone full-stack web application. It is composed of three primary layers:
•	Frontend: A React.js + Tailwind CSS interface for all user roles
•	Backend: A FastAPI (Python) server exposing RESTful API endpoints
•	AVRE Engine: A modular scoring component (rule-based or ML-powered) that ranks vendors

In future iterations, AVRE may integrate with external map APIs (OpenStreetMap / Leaflet.js) and notification services.

2.2 Product Functions Summary
Module	Core Functions
User Module	Register, Login, Create request, View matches, Accept vendor, View history
Vendor Module	Register, Login, Manage inventory, Set response time, Accept/reject requests
Admin Module	View all users/vendors/requests, Remove fake entries, Monitor stats
AVRE Engine	Fetch eligible vendors, Compute relevance score, Return ranked list

2.3 User Classes and Characteristics
User Role	Description	Technical Proficiency
Requester	Person in need of a resource. May be in an emergency situation. Needs simple, fast interface.	Low to Medium
Vendor	Shop owner or supplier. Registers resources and responds to match requests.	Low to Medium
Admin	Platform operator who monitors all system activity and manages data quality.	Medium to High

2.4 Operating Environment
•	Web Browser: Chrome, Firefox, Edge (latest 2 versions)
•	Mobile: Responsive layout; no native app required for MVP
•	Backend: Python 3.10+, FastAPI
•	Database: SQLite (MVP), PostgreSQL (production)
•	Hosting: Frontend on Vercel, Backend on Render

2.5 Design and Implementation Constraints
•	MVP must be functional within a 4-week development cycle
•	No paid third-party APIs in MVP phase
•	ML model is optional for MVP; rule-based scoring is the baseline
•	System must be deployable as a single backend service

2.6 Assumptions and Dependencies
•	Vendors accurately self-report their inventory and response times
•	All users have access to a modern web browser and internet connection
•	Location data is provided manually (GPS integration is optional)
•	Scikit-learn will be used if the ML scoring module is implemented

3.5 AVRE Matching Engine
The AVRE Engine is the core intelligence layer of the platform. It ranks vendors based on a weighted relevance score computed at the time of each request.

3.5.1 Scoring Formula
The relevance score is calculated as:

Score = (w1 × distance_score) + (w2 × stock_score) + (w3 × rating_score) + (w4 × speed_score) + (w5 × urgency_score)

3.5.2 Default Scoring Weights
Factor	Weight	Description
Distance Score	35%	Proximity of vendor to requester location — closer vendors score higher
Stock Score	20%	Availability and sufficient quantity of the requested resource
Rating Score	15%	Cumulative vendor rating from historical fulfilled requests
Speed Score	15%	Vendor's average response time — faster vendors score higher
Urgency Score	15%	Adjusts ranking when request urgency level is High or Critical

Req ID	Requirement	Priority
FR-501	The system shall fetch all vendors whose category matches the requested resource type.	High
FR-502	The system shall compute a weighted relevance score for each eligible vendor.	High
FR-503	The system shall return vendors sorted by descending relevance score.	High
FR-504	The system shall exclude vendors whose is_active flag is false.	High
FR-505	The system shall exclude vendors with insufficient stock for the requested quantity.	High
FR-506	The system shall support admin-configurable scoring weights.	Low
FR-507	The system shall optionally support a machine learning model (Random Forest, XGBoost, or Gradient Boosting) for score prediction.	Low
